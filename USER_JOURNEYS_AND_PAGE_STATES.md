# 用户旅程与页面呈现说明

本文档描述 **不同身份 / 状态下的用户** 在关键入口 **点击后看到什么**，以及 **如何引导信息补全**。实现细节以代码为准；路由与跳转详见 `[PAGES_GUIDE.md](./PAGES_GUIDE.md)`，主流程见 `[PROJECT_REQUIREMENTS.md](./PROJECT_REQUIREMENTS.md)` §3。

---

## 1. 用户状态速查


| 状态维度     | 取值说明                                                             |
| -------- | ---------------------------------------------------------------- |
| **登录**   | 未登录 / 已登录（本地持久化，刷新仍有效）                                           |
| **本地线索** | `savedLeadsStorage`：待签约线索、已转项目（已签约）                              |
| **全局表单** | `FormData`：风格测评、深度测评、合同签署等是否已填写（与演示用本地状态同步）                      |
| **入口来源** | 欢迎页、项目列表、需求书补全（`from=requirements`）、合同页（`leadId` / `from=home`）等 |

## 2. 核心入口：用户点了什么 → 去哪 → 页面上是什么

### 2.1 欢迎页 `/`


| 用户情况    | 点击         | 结果页面与主要内容                                                  |
| ------- | ---------- | ---------------------------------------------------------- |
| 任意      | **立即开启体验** | → `/style-eval`：10 道风格题 + 1 页结果（风格画像等）                     |
| **未登录** | **进入我的项目** | → `/login`，登录成功默认去 `**/projects`**（若带 `redirectTo` 则去指定路径） |
| **已登录** | **进入我的项目** | → `**/projects`**（见下文「项目列表呈现」）                             |
| 任意      | 左下角 **设置** | → `/admin` 后台管理入口                                          |


### 2.2 风格测评完成后（结果页）


| 用户操作        | 去向与页面内容                                                   |
| ----------- | --------------------------------------------------------- |
| **去填写项目概况** | → `/leads` 步骤 1：项目概况                                      |
| **返回首页**    | → `/` 欢迎页（**未注册、未把测评与账号绑定**；若之后单独注册，测评数据是否保留取决于当前实现与存储策略） |


### 2.3 项目列表 `/projects`（「我的项目 / 欢迎回家」）

**进入方式**：欢迎页「进入我的项目」（已登录）、深度测评完成跳转、登录成功默认页、合同线索流「返回主页」到列表等。


| 本地数据情况                       | 页面呈现（当前实现）                                                                                                                                |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **无任何已签约项目**（`converted` 为空） | 标题「欢迎回家」+ 副文案；**不显示**「已签约 / 待签约」双 Tab；主区域为 **待签约线索**：有「新增线索」；若 `pending` 为空则 **空状态**：「暂无线索」+ 引导去 **新增线索**（`/leads?from=projects`，说明可不再注册） |
| **已有至少一个已签约项目**              | 双 Tab：**已签约项目** / **待签约线索**；已签约卡片点击进入 `**/home`** 并绑定该线索工作台；待签线索 **去签署** → `/contracts?leadId=…`                                          |


**产品含义**：  

- 用户「做完风格测评就关掉 → 后来注册/登录 → 点我的项目」：若 **从未在本机保存过线索或走完注册+线索合同流**，列表里往往是 **空线索**；**不会**自动出现「刚做的测评」 unless 数据已写入同一用户上下文。  
- 引导：**新增线索** 或从 **风格测评 → 线索 → 注册 → 合同** 主链路把项目跑通。

### 2.4 主链路：风格测评 → 项目中心（摘要）

1. `/` → 风格测评 → `/leads`（两步）→ `/budget-breakdown`（饼图）→ **注册** → `/contracts`（签意向金 + 付款页）→ **深度测评** → `**/projects`** → 点项目 → `**/home` 项目中心**。
2. 项目中心侧栏：**项目协议、用户需求、预算资金、订单方案**（「待定决策」在订单方案页顶部：**仅龙湖璟宸府演示项目**会列出待办；其他项目显示「当前暂无待处理事项」）；订单可进详情、报价、结算、设计反馈等（见 `PAGES_GUIDE`）。

### 2.5 从项目中心「用户需求」补全（顾问已录入概况）

- 需求书 **全部为空** 时：提示「暂无需求内容」，引导 **从风格测评开始** → `/style-eval?from=requirements`；结果页 **开始深度需求测评** → `/deep-eval?from=requirements&leadId=…`（**不经过**线索收集、注册、合同；测评目录亦隐藏线索/合同步骤）。  
- 完成后回到项目中心并聚焦「用户需求」Tab。

### 2.6 合同签署不同来源（用户看到的主按钮不同）


| 来源                    | 用户感知                                                        |
| --------------------- | ----------------------------------------------------------- |
| **注册后**（无 `leadId`）   | 付款页复制后：**完成支付进入深度测评** → `/deep-eval`；返回主页 → 欢迎页             |
| **项目中心**（`from=home`） | 付款页：**进入我的项目** → `/home`；不强制进深度测评                           |
| **待签线索**（`leadId`）    | 付款页复制后：**进入我的项目** + **继续深度测评**（带 `leadId`）；返回 → `/projects` |

## 3. 典型场景叙事（便于产品/客服对照）

### 场景 A：新用户从风格测评一路到项目中心

- 欢迎页 → 测评 → 线索两步 → 预算拆解 → 注册 → 合同两步 → 深度测评 19 步 → **项目列表** 出现已转项目卡片 → 进入 **项目中心**，可看协议、需求汇总、预算 Sankey、订单等（数据随表单与 Mock 填充情况变化）。

### 场景 B：测评完点「返回首页」，之后再注册并点「我的项目」

- **当前页**：欢迎页；**注册成功** 通常进合同流，不一定回到测评结果。  
- **登录后打开「我的项目」**：见 §2.3 —— 若本地无已保存线索/未完成线索侧合同转化，多为 **空线索列表**；需引导 **新增线索** 或重新走 **测评 + 线索** 以关联项目。  
- **信息完整性建议**：在结果页或登录后首次进 `/projects` 用文案说明「完成线索与签约后，项目会出现在此」。

### 场景 C：老用户只登录、从未在本设备做过测评

- `/projects`：可能仅空状态或仅有历史线索；进 `/home` 后需求书、订单等依赖绑定线索与全局数据。  
- 引导：新增线索 / 从菜单进风格测评与深度测评。

### 场景 D：销售录入项目，客户只补测评 + 深度测评

- 从项目中心需求书入口进入缩短路径（§2.5），保证 **风格 + 深度** 与当前 `leadId` 绑定。

## 4. 信息完整性：建议引导矩阵


| 信息块     | 主要采集路径                   | 用户可能缺失时的引导                                  |
| ------- | ------------------------ | ------------------------------------------- |
| 风格画像    | `/style-eval`            | 需求书空 → 「从风格测评开始」；项目列表空 → 主 CTA 体验链路         |
| 项目概况与联系 | `/leads`                 | `/projects` 新增线索；主链路测评后自动进入                 |
| 注册与合同   | `/register`、`/contracts` | 待签约线索卡片「签署合同」                               |
| 深度需求    | `/deep-eval`             | 注册流付款后默认进入；线索流付款页「继续深度测评」                   |
| 需求书全文   | 上述数据汇总                   | 项目中心「用户需求」查看与提交确认                           |
| 预算/订单   | 工作台、订单状态                 | 依赖订单 Mock/后端；见 `ORDER_STATE_TRANSITIONS.md` |


**原则**：  

1. **列表空** → 明确下一步是「新增线索」还是「从首页开始体验」。
2. **有待签** → 优先「签署合同」再进工作台或深度测评。
3. **已进工作台但需求空** → 缩短路径 `from=requirements` 补测评。
4. **登录态** 与 **线索本地存储** 在演示环境以浏览器为准，上线需与账号维度对齐，避免「注册了却看不到测评」的困惑。
