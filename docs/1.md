
# Dsphr Home 平台 — 项目需求文档

## 1\. 文档信息

| 项目 | 说明 |
| :--- | :--- |
| **文档名称** | Dsphr Home 项目需求文档 |
| **文档用途** | 描述家装设计服务平台的业务需求、功能范围、数据模型与交互流程，供产品、设计与开发统一参考 |

-----

## 2\. 背景

Dsphr Home 面向家装客户，定位为 全案项目服务平台。平台需要系统化采集客户偏好与项目信息，以项目为中心管理订单、合同、预算与交付进度，并支持报价与结算、设计反馈等功能，形成从线索到交付的完整服务链路。

-----

## 3\. 目标

全案项目客户可以通过产品清晰的了解整个项目及每一个订单的情况，并实现以下功能：

  - **需求采集**：通过风格测评、线索收集、深度需求测评，系统化采集客户偏好与项目信息，形成可用的项目需求书与预算依据。
  - **项目与订单管理**：以项目为中心，管理多订单、合同、预算与进度，支持订单状态流转与待定决策展示。
  - **报价与结算**：提供订购报价（V1 起）、交付报价、最终结算单的展示、签字与反馈；意向阶段以方案反馈/确认为主，不提供「意向报价单」。
  - **设计反馈与维保**：支持设计方案/图纸的查看与反馈，以及订单交付后的维保申请入口。

-----

## 4\. 范围

### 4.1 本需求包含

  - **功能模块与页面**：欢迎页、家居风格测评、线索收集、初步预算拆解、注册/登录、合同签署、深度需求测评、项目列表、项目中心、用户需求、预算资金、订单方案、订单详情、设计反馈、报价单、结算单；
  - **欢迎页**：放于官网首页展示，客户可通过官网点击进行风格测评/进入我的项目。
  - **家居风格测评**：7个问题对应7个维度，用户通过测评能拿到测评卡片，需要打通CIL读取问题数据 and 图片数据。
  - **线索收集**：基于CRM的需要采集用户线索信息，并填写进CRM。
  - **初步预算拆解**：基于用户输入的信息，调取CRM算法，生成一个初步EPC预算拆解卡片，用户可保存。
  - **注册/登录**：用户通过短信验证码的形式注册和登录。
  - **合同签署**：用户可查看合同并在线签字签署合同，合同签署后可复制账号信息进行转账。
  - **深度需求测评**：通过一系列项目相关的问题，对项目的需求做深入的了解，深度测评的信息最终会生成一个用户需求书。
  - **项目列表**：用户名下的项目和线索展示，该页面可供用户增加线索。
  - **项目中心**：具体某一个项目的管理中心，包含项目协议、用户需求、预算资金、订单方案等板块，用户可直观浏览项目和订单信息，并进行决策和反馈。
  - **用户需求**：用户的需求书，作为项目展开一切的“宪法”，可进行编辑，也可以进行确认，每一次修改都会有历史的修改存档和快照。
  - **预算资金**：展示项目预算、订单分配、资金管理的内容，核心是2个预算桑基图的展示，1个是EPC预算拆解桑基图、一个是订单预算拆解桑基图。
  - **订单方案**：用于展示订单，客户对于订单的待定决策，桑基图订单拆解的展示，以及订单列表。
  - **订单详情**：订单的详情页，包含订单状态、方案、报价单、结算单等相关信息列表，点击可进入子页面。
  - **设计反馈**：客户可进入该界面查看订单设计方案，可评论反馈、可通过方案，可查看历史版本。
  - **报价单/结算单**：客户可查看报价单/结算单，给予通过确认或者输入评论反馈。报价单/结算单的明细数据从PSO读取。
  - **状态跳转**：客户的一些动作会驱动订单状态机的跳转。
  - **PSO 对接**：项目订单的数据需要和PSO交互对接，支持读写。

### 4.2 本需求不包含

  - 目前仅支持一个项目一个授权决策人可点击查看和反馈，不支持一个项目多个用户查看。
  - 订单详情页不做选品的展示。
  - 不做签字记忆复用的功能。

## 5\. 核心概念

  - **线索**：客户填写的项目概况与联系信息，采集后写入 CRM；线索可转为项目，合同签署后线索可转变为项目。
  - **项目**：一个全案项目的总称，下挂多笔订单；项目中心即该项目的管理视图，包含项目协议、用户需求、预算资金、订单方案等板块。
  - **订单**：项目下的具体交付单元，有状态流转；订单详情页展示订单状态、方案、报价单、结算单等，客户可进入设计反馈、报价/结算子页面。
  - **订单状态与阶段**：订单在生命周期内经历意向期、订购期、交付期、验收期、维保期等阶段；不同的阶段对应的状态不一样。具体状态码与流转见7.5
  - **用户需求书**：由风格测评、线索、深度需求测评汇总而成，作为项目展开的“宪法”；可编辑、确认，每次修改有历史存档与快照，在项目中心「用户需求」板块展示。
  - **预算拆解（EPC）**：基于用户输入信息、通过 CRM 算法生成的 E/P/C 初步预算；在「预算资金」中通过 EPC 预算桑基图、订单预算桑基图展示项目预算与订单分配。
  - **待定决策**：客户在订单上待完成的决策项（如方案确认、设计反馈、报价/结算签字等），在订单方案板块展示，引导客户进入设计反馈或报价单/结算单操作。
  - **报价单与结算单**：包括订购报价（V1 起）、交付报价、最终结算单；按订单阶段展示，客户可查看、通过确认或输入评论反馈；明细数据从 PSO 读取。意向阶段不提供「意向报价单」。
  - **合同签署**：客户在线阅读意向金合同并在线签字，签署后可复制对公账号信息完成转账；合同入口包括注册后、从项目中心进入、从线索「去签署」进入，步骤 2 完成后根据来源进入深度测评或项目中心。
  - **PSO**：项目与订单数据的系统对接方，项目/订单及报价单、结算单明细等与 PSO 交互，支持读写。
  - **CRM**：线索及初步预算算法等数据的对接方，线索信息写入 CRM，初步预算拆解调 CRM 算法。
  - **CIL**：风格测评题目与图片等内容的读取来源，需打通以拉取测评题目与图片数据。

-----

## 6\. 用户角色

  - **C 端用户（客户/授权决策人）**：当前为一个项目一个授权决策人，可完成风格测评、线索填写、注册/登录（短信验证码）、合同签署、深度测评；在项目中心查看项目与订单、用户需求、预算资金，处理待定决策；在订单详情查看方案、报价单、结算单并进行设计反馈或维保申请。
  - **后台/管理**：通过 admin 等入口进行测评目录导航与流程跳转（视实际需要开放）。

## 7\. 系统整体逻辑

### 7.1 主流程概览

  - **新用户**：官网欢迎页 → 风格测评（CIL 题目与图片）→ 线索收集（写入 CRM）→ 初步预算拆解（CRM 算法，可保存）→ 注册/登录（短信验证码）→ 深度需求测评（生成用户需求书）→ 合同签署（意向金签字 + 对公转账）→ 项目列表 → 进入项目中心 → 项目协议 / 用户需求 / 预算资金 / 订单方案（含待定决策）→ 订单详情 → 报价单 / 结算单 / 设计反馈。
  - **老用户**：欢迎页「进入我的项目」→ 未登录则登录 → 项目列表 → 选择项目 → 项目中心；从项目中心进入合同时。

### 7.2 数据与状态

  - 风格测评、线索、深度测评等采集数据汇总为用户需求书，在项目中心展示并可编辑与确认，有修改历史与快照。
  - 订单及报价单、结算单与 PSO 对接，订单状态驱动报价单/结算单可见性及客户可执行操作；待定决策由订单状态与客户待办推导，在订单方案板块展示。具体状态与流转见 §5 核心概念 及 `ORDER_STATE_TRANSITIONS.md`。
  - 用户需求书与深度测评步骤的区块对应关系见 DEEP\_EVAL\_TO\_REQUIREMENTS\_MAPPING.md。

### 7.3 系统对接

  - **CRM**：线索提交与写入；初步预算拆解算法调用。
  - **PSO**：项目、订单数据读写；报价单、结算单明细读取；客户动作驱动的订单状态跳转。
  - **CIL**：风格测评题目与图片数据读取。
  - **auth**：注册、登录（短信验证码）等认证。

### 7.5 订单状态及流转

#### 7.5.1 完整状态字典 (State Dictionary)

| 业务阶段 | 状态码 | 状态名称 | 核心逻辑与功能说明 |
| :--- | :--- | :--- | :--- |
| **1. 意向期** | S00 | 方案筹备中 | 初始状态；禁用意向报价单功能。 |
| | S01 | 意向沟通中 | 方案沟通阶段；无意向报价单；确认后转深化。 |
| | S05 | 客户决策中 | 自动流转：S01 逾期未决策后自动触发。 |
| **2. 订购期** | S02 | 订单深化中 | 包含设计深化与精准报价的汇总状态。 |
| | S02-01 | 提案设计中 | 专注于方案深化与图纸绘制。 |
| | S02-02 | 订购报价中 | 进入精准报价核算阶段。 |
| | S03 | 订购确认中 | 方案/报价已提交，等待客户最终确认。 |
| **3. 交付期** | S06-01 | 交付设计中 | 施工图、节点图等详细技术设计。 |
| | S06-02 | 方案汇报中 | 最终施工方案汇报。 |
| | S06-03 | 交付备货中 | 材料采购、供应链及工人调度。 |
| | S06-04 | 交付施工中 | 正式进场，按施工节点推进。 |
| | S06-05 | 交付内审中 | 施工完成后的内部质量预检。 |
| | S13 | 订单休眠中 | 异常/挂起：交付暂停，前端时间轴特殊展示。 |
| **4. 验收期** | S07 | 订单验收中 | 提交竣工验收及最终结算单，等待确认。 |
| | S09 | 订单整改中 | 客户异议触发，进入内部整改流程。 |
| **5. 维保期** | S11 | 订单已交付 | 验收通过且尾款结清，正常完结。 |
| | S10 | 订单维保中 | 进入售后阶段，处理维修或保养申请。 |
| | S12 | 订单已结束 | 终态（由交付/验收/维保流转而来）。 |
| **6. 独立终止** | S04 | 客户已婉拒 | 意向阶段流失，非成交关闭。 |
| | S08 | 订单终止中 | 过程中途异常终止（违约/协商一致）。 |

#### 7.5.3 状态流转路径 (State Transition Paths)

##### 7.5.3.1 正常正向流转 (Normal Forward Flow)

这是订单顺利推进的标准路径。

  - **S00 -\> S01 (筹备转沟通)**
      - 触发动作: 方案/沟通材料就绪，客户可进入意向沟通。
  - **S01 -\> S02-01 (意向转提案设计)**
      - 触发动作（产品/后台语义）：意向沟通完成、方案确认后进入深化；当前 C 端已取消「意向报价单」，意向阶段以方案反馈为主，具体跳转以后台/workflow 为准。
  - **S02-01 -\> S02-02 (提案设计转订购报价)**
      - 触发动作: 设计师完成深化方案和详细图纸，进入精准报价阶段。
  - **S02-02 -\> S03 (订购报价转订购确认)**
      - 触发动作: 设计师完成精准报价，在后台点击“提交客户确认”。
  - **S03 -\> S06-01 (订购转交付设计)**
      - 触发动作: 客户在订单详情页查看“最终报价单”，点击“确认报价单”并完成签名。
  - **S06-01 -\> S06-02 (设计转汇报)**
      - 触发动作: 设计师完成所有施工图纸，在后台点击“发起方案汇报”。
  - **S06-02 -\> S06-03 (汇报转备货)**
      - 触发动作: 客户在详情页浏览完所有图纸，点击“完成审阅”并同意方案。
  - **S06-03 -\> S06-04 (备货转施工)**
      - 触发动作: 施工经理在后台确认材料和人员就位，点击“进场开工”。
  - **S06-04 -\> S06-05 (施工转内审)**
      - 触发动作: 施工经理完成所有施工节点，在后台点击“发起内部审核”。
  - **S06-05 -\> S07 (内审转验收)**
      - 触发动作: 内部审核通过，提交竣工验收及最终结算单，等待客户确认。
  - **S07 -\> S11 (验收通过交付)**
      - 触发动作: 客户在订单详情页查看“订单结算单”，点击“确认结算单”并完成签名。

##### 7.5.3.2 逆向/打回流转 (Reverse / Rejection Flow)

当客户对方案、报价或施工质量不满意时触发的打回路径。

  - **S01 -\> S04 (意向转婉拒)**
      - 触发动作: 客户对方案或沟通结果不满意，明确表示拒绝。
  - **S03 -\> S02-01 / S02-02 (订购打回提案/报价)**
      - 触发动作: 客户在“订购确认中”阶段，点击底部的“希望调整方案”并提交反馈。订单退回提案设计或订购报价阶段重新修改。
  - **S06-02 -\> S06-01 (方案打回设计)**
      - 触发动作: 客户在“方案汇报中”阶段，浏览图纸后提出修改意见，点击“提交反馈”。订单退回交付设计阶段。
  - **S06-05 -\> S06-04 (内审不合格打回施工)**
      - 触发动作: 内部审核发现问题，打回施工阶段进行整改。
  - **S07 -\> S09 (验收不合格整改)**
      - 触发动作: 客户在“订单验收中”阶段，对施工质量或结算单有异议，点击“对结算有异议”并提交反馈。
  - **S09 -\> S07 (整改完成转验收)**
      - 触发动作: 施工团队完成整改，重新提交验收申请和更新后的结算单。

##### 7.5.3.3 异常/超时/终止流转 (Exception / Timeout / Termination Flow)

订单在未正常交付前被提前结束，或因客户未及时操作而进入特殊状态。

  - **S01 -\> S05 (意向超时转决策中)**
      - 触发动作: S01阶段设定的时间到期，客户迟迟未做反馈，系统自动流转至S05。
  - **任意状态 (S01\~S07) -\> S08 (订单终止)**
      - 触发动作: 客户在订单详情页右上角点击红色的“终止订单”按钮，或后台管理员强制终止。
      - 说明: 终止后，订单进入冻结状态，不可再进行常规的正向流转，通常需要走线下清算流程。

##### 7.5.3.4 售后/维保流转 (Post-sales / Maintenance Flow)

订单正常交付后发生的售后服务。

  - **S11 -\> S10 (发起售后维保)**
      - 触发动作: 客户在“订单已交付”状态下，在详情页右上角点击蓝色的“申请维保”按钮，填写维保需求。
  - **S10 -\> S11 (维保完成)**
      - 触发动作: 维保团队完成维修服务，客户确认修复完毕，订单状态恢复为“已交付”。

### 7.6 报价单与结算单状态生命周期 (Status Lifecycle)

#### 7.6.1 报价单与结算单状态

| 状态 | 描述 | 触发动作 (驱动变化) |
| :--- | :--- | :--- |
| **未查看未签字** | 初始状态，单据已生成 | 无 (初始状态) |
| **已查看未签字** | 客户已阅读单据 | 客户打开单据详情页 |
| **已查看已反馈** | 客户已提交修改意见 | 客户点击“希望调整方案”并提交反馈 |
| **已查看已签字** | 客户已确认单据 | 客户点击“确认单据”并完成签名 |

#### 7.6.2 报价单与结算单状态变化

  - 未查看未签字 --(客户查看)--\> 已查看未签字
  - 已查看未签字 --(客户签字)--\> 已查看已签字
  - 已查看未签字 --(客户反馈)--\> 已查看已反馈
  - 已查看已反馈 --(设计师/后台更新单据)--\> 未查看未签字 (新单据生成)

**注**：在“已查看已反馈”状态下，客户无法进行签字操作，必须等待后台根据反馈更新单据后，重新进入新的“未查看未签字”状态。

#### 7.6.3 视觉反馈规范 (Color Coding)

  - **红色 (Danger)**: 用于 `未查看未签字`，提示客户有新单据待处理。
  - **橙色 (Warning)**: 用于 `已查看未签字`，提示客户需尽快完成签字。
  - **绿色 (Success)**: 用于 `已查看已签字`，表示流程已闭环。
  - **蓝色 (Info)**: 用于 `已查看已反馈`，表示单据正在处理反馈意见。

## 8.6 数据流简要说明

主链路顺序（与 `Question Number.md` 一致）：**风格测评 → 线索收集 → 初步预算拆解 → 注册 → 深度测评 → 合同签署 → 项目列表 → 项目中心**。

1.  **家居风格测评**：主链路第一步，q1～q7 + result 产出风格画像、色彩基因及硬装/软装建议，保存至 VoyageCard；q8/q9/q10 已移入深度测评（在 Q2-5 与 Q2-6 之间），设计上流入需求书「风格人设」「设计定位」，当前尚未打通。
2.  **线索收集**：DE-1、DE-2 产出项目名、类型、面积、预算、城市及业主联系方式等，对应需求书「项目概览」。
3.  **初步预算拆解 / 注册 / 合同签署**：不直接写入需求书区块；合同在深度测评之后，签署完成后进入项目列表与项目中心。
4.  **深度测评**：按 Q2-4 → Q2-5 → q8/q9/q10 → Q2-6～Q2-21 顺序采集房型、房屋现状、成员、空间与设备等，对应需求书各区块。
5.  **项目需求书**：在项目中心侧栏以「项目需求书」形式展示，汇总上述所有采集结果（含风格色彩），供用户查看、编辑与提交。
6.  **编辑模式**：需求书支持编辑，可补充户型图、现场视频、智能家居选项、特殊设备、风水、收纳、个性化说明等。
