# Dsphr Home 平台 — 项目需求文档

本文档基于当前代码实现整理，描述 **Dsphr Home** 家装设计服务平台的业务需求、功能范围、数据模型与交互流程，供产品、设计与开发统一参考。

---

## 1. 项目概述

### 1.1 产品定位

Dsphr Home 是一款面向家装客户的 **设计服务与项目协作平台**，核心能力包括：

- **需求采集**：通过风格测评、线索收集、深度需求测评，系统化采集客户偏好与项目信息。
- **项目与订单管理**：以项目为中心，管理多订单、合同、预算与交付进度。
- **报价与结算**：意向报价、订购报价、最终结算单的展示、签字与反馈。
- **设计反馈与维保**：设计方案/图纸的查看与反馈，订单交付后的维保申请。

### 1.2 技术栈（当前实现）

| 类别 | 技术 |
|------|------|
| 前端框架 | React 19 + TypeScript |
| 路由 | react-router-dom v7 |
| 构建 | Vite 6 |
| 样式 | Tailwind CSS 4 |
| 状态 | React Context（GlobalContext、DeepEvalFormContext） |
| 后端/API | 前端对接 leads、auth、projects、fundPool、designVoyage 等 service 层 |

### 1.3 用户角色

- **C 端用户（客户）**：完成风格测评、线索填写、注册/登录、合同签署、深度测评、查看项目/订单/报价/结算、设计反馈。
- **后台/管理**：通过 `/admin` 入口进行测评目录导航与流程跳转（当前为演示用后台入口）。

## 3. 核心业务流程

### 3.1 新用户从「风格测评」到「项目中心」的主流程

1. **欢迎页** → 点击「立即开启体验」→ **家居风格测评**（`/style-eval`）  
2. 测评完成 → 「去填写项目概况」→ **线索收集**（`/leads`）  
3. 线索步骤 2 完成 → **初步预算拆解**（`/budget-breakdown`）→ 可保存、可点击注册  
4. **注册**（`/register`）→ 注册成功 → **合同签署**（`/contracts`）  
5. 合同步骤 1：意向金合同阅读与电子签署；步骤 2：对公转账信息，复制后「完成支付进入深度测评」  
6. **深度需求测评**（`/deep-eval`）19 步完成 → **项目列表**（`/projects`）  
7. 在项目列表中点击项目 → **项目中心**（`/home`）  
8. 项目中心内可进入：**项目需求书**、**预算看板**（`/budget`）、**项目订单**、**项目合同**；从订单进入 **订单详情** → **报价单/结算单/设计反馈** 等。

### 3.2 老用户 / 从项目中心进入的流程

- 欢迎页「进入我的项目」→ 未登录则 **登录**（`/login`）→ 登录后若仅一个项目则直接 **项目中心**，否则 **项目列表** → 选项目 → **项目中心**。
- 从项目中心进入 **合同**（`?from=home`）：步骤 2 主按钮为「进入我的项目」，不再进入深度测评；「返回主页」指返回项目中心。

### 3.3 合同签署逻辑

- **已签署**：若 `contractAccepted` 且存在 `contractSignatureData`，进入合同页自动到步骤 2（付款信息）；步骤 1 可显示「已签署，查看付款信息」。
- **来源区分**：无 `from` 视为注册流程；`from=home` 视为从项目中心进入，上一步/返回均为项目中心。

## 4. 数据与状态需求

### 4.1 全局表单与用户状态

- **FormData**（`src/types.ts`）：承载风格测评、线索、深度测评、合同等全部采集字段；包括但不限于：
  - 用户/项目基础：userName、userPhone、userCity、projectName、projectType、projectArea、houseType、houseCondition、budgetStandard 等；
  - 深度测评：房型、房屋现状、成员、空间、习惯、设备、风水、智能家居、适老、旧物、其他需求等（Q2-4～Q2-21 对应字段）；
  - 风格：styleId、styleName、colorGene、styleSuggestions；
  - 合同：contractAccepted、contractSignatureData、contractCustomText；
  - 订单列表：orders（初始来自 `mockOrders`）。
- **GlobalContext**：提供 `data: FormData`、`updateData`、`setData`、`isLoggedIn`、`setLoggedIn`，供全局读写。
- **DeepEvalFormContext**：为深度测评/需求书等提供表单上下文。

### 4.2 订单模型与状态

- **订单字段**（结合 `mockOrders` 与订单相关页面）：id、projectId、title、status、contains、date、amount、isInteractive 等。
- **订单状态**：以 `docs/ORDER_STATE_TRANSITIONS.md` 为准，包括但不限于：
  - 意向期：S00、S01、S05；
  - 订购期：S02、S02-01、S02-02、S03；
  - 交付期：S06-01～S06-05、S13；
  - 验收期：S07、S09、S08；
  - 维保期：S11、S10；
  - 终止/结束：S04、S12。
- **状态与阶段颜色**：由 `orderStatus.ts` 映射（intention/ordering/delivery/acceptance/maintenance/red/gray），用于列表与详情页展示。

### 4.3 订单状态流转与单据可见性

- **意向报价单**：订单状态 &gt; S00 时可见；S01 时客户未签字，签字后可从 S01 进入 S02。
- **订购报价单**：订购期及之后可见；S03 时为「未查看未签字」或「已查看已反馈」或「已查看未签字」，客户签字后进入 S06。
- **交付报价单**：交付期及之后可见。
- **最终结算单**：验收期及之后可见；S07 时为「未查看未签字」或「已查看已反馈」或「已查看未签字」，签字后进入 S11。

报价单/结算单的四种展示状态：**未查看未签字**、**已查看未签字**、**已查看已反馈**、**已查看已签字**；流转规则见 `ORDER_STATE_TRANSITIONS.md`。

### 4.4 项目需求书与数据来源

- **项目需求书**：在项目中心侧栏以「项目需求书」形式展示，汇总风格测评、线索、深度测评的采集结果。
- **区块结构**（见 `DEEP_EVAL_TO_REQUIREMENTS_MAPPING.md`）：项目概览、结构现状（设计定位）、项目图纸与视频、成员画像、全屋需求（系统设备、智能家居、全屋设备、风水、收纳、个性化）、空间需求（客厅、餐厅、厨房、卫生间等）。
- 需求书与 FormData / 深度测评步骤的对应关系已在该文档中按步骤 ID 与区块列出；风格人设、设计定位与风格测评的打通为已知待完善项。

## 5. 服务层与接口需求

当前前端已对接或预留的 service 层（实现以代码为准）：

| 服务 | 职责 |
|------|------|
| auth | 登录、注册等认证相关 API |
| leads | 线索提交、查询等 |需要和飞书crm打通
| projects | 项目列表、项目详情等 |需要和飞书pso打通
| fundPool | 资金池/预算相关 |需要和飞书pso打通
| designVoyage | VoyageCard（风格测评等）、Leads 选项、Leads 提交等 |

## 6. 非功能需求（基于实现归纳）

### 6.1 体验与交互

- **测评目录**：在 admin、style-eval、leads、contracts、deep-eval 等页提供统一测评目录（基于 `navigationConfig.ts`），支持按步骤跳转。
- **返回与上一步**：各多步流程需明确「返回」目标（上一题、上一页、欢迎页、项目中心等），与 PAGES_GUIDE 一致。
- **视觉规范**：订单阶段颜色、报价单/结算单状态颜色等需与设计规范一致（见 ORDER_STATE_TRANSITIONS 与主题变量）。

### 6.2 开发规范（来自 INSTRUCTIONS.md）

- **单页修改原则**：单次修改尽量限定在单页或单组件及其私有子组件，不随意改动路由、全局 Context 或无关页面。
- **逻辑保留原则**：不随意变更既有业务逻辑与 API 结构；若 UI 变更导致逻辑变更需说明并确认。
- **沟通语言**：开发与需求讨论使用中文。
- **NN 前缀**：用户消息以 NN 开头时仅做文字回复，不执行代码编辑。

### 6.3 兼容与扩展

- 订单状态与阶段、财务单据显示逻辑需与后端一致，便于后续接入真实订单与工作流。
- 项目需求书需支持从真实采集数据动态渲染，并保留「展示页」示例能力。
