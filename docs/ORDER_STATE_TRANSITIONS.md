# 订单状态流转全景说明 (Comprehensive Order State Transitions)

本文档记录了系统中所有可能的订单状态（Order Status）及其完整的流转路径和触发条件。

## 一、 完整状态字典 (State Dictionary)

| 状态码 | 状态名称 | 阶段说明 |
| :--- | :--- | :--- |
| **S00** |方案筹备中 | 订单刚创建，设计师/管家筹备方案与沟通材料，**不提供意向报价单** |
| **S01** | 意向沟通中 | 客户进入方案与意向沟通，**无意向报价单**；确认方案后进入深化/订购流程 |
| **S02** | 订单深化中 | 方案深化设计、详细图纸绘制、精准报价阶段 |
| **S02-01** | 提案设计中 | 方案深化设计、详细图纸绘制 |
| **S02-02** | 订购报价中 | 精准报价阶段 |
| **S03** | 订购确认中 | 向客户提交最终深化方案和精准报价，等待客户确认 |
| **S04** | 客户已婉拒 | 客户觉得报价或者方案不满意，决定不成交 |
| **S05** | 客户决策中 | 当到达S01的时间到期了，那么自动跳到S05，意味着客户迟迟未决策|
| **S06-01** | 交付设计中 | 正式进入交付阶段，进行施工图、节点图等详细设计 |
| **S06-02** | 方案汇报中 | 向客户汇报最终施工方案及图纸 |
| **S06-03** | 交付备货中 | 方案通过后，进行材料采购、工人调度等准备工作 |
| **S06-04** | 交付施工中 | 正式进场施工，按节点推进 |
| **S06-05** | 交付内审中 | 施工完成，现场进行内部审核验收|
| **S07** | 订单验收中 | 施工完成，提交竣工验收及最终结算单，等待客户确认 |
| **S08** | 订单终止中 | 订单因各种原因（违约、协商一致等）提前终止 |
| **S09** | 订单整改中 | 客户对验收或结算有异议，进行内部整改 |
| **S10** | 订单维保中 | 交付后，客户发起售后维修或保养申请 |
| **S11** | 订单已交付 | 验收通过，结清尾款，订单正常完结 |
| **S12** | 订单已结束 | 生命周期终结（多从交付/验收/维保流转），详见 §2 独立终止状态 |
| **S13** | 订单休眠中 | 交付阶段暂停/休眠；**订单详情页时间轴**中与 S04/S05/S08 等同列为异常展示逻辑（见 `OrderDetailPage`） |

## 二、 订单阶段与状态映射 (Order Phases and States Mapping)

根据系统设计，订单的生命周期被划分为 5 个主要阶段（Phase）以及 2 个全局终止状态。以下是具体的对应关系：

### 1. 意向期 (Intention Phase)
* **S00-方案筹备中**
* **S01-意向沟通中**
* **S05-客户决策中**

### 2. 订购期 (Ordering Phase)
* **S02-订单深化中**
* **S03-订购确认中**

### 3. 交付期 (Delivery Phase)
* **S06-订单交付中**
* **S13-订单休眠中**

### 4. 验收期 (Acceptance Phase)
* **S07-订单验收中**
* **S09-订单整改中**
* **S08-订单终止中**

### 5. 维保期 (Maintenance Phase)
* **S11-订单已交付**
* **S10-订单维保中**

### 独立终止状态 (Terminal States)
* **S04-客户已婉拒** (主要从意向期或订购期流转，代表前期流失)
* **S12-订单已结束** (主要从交付期、验收期或维保期流转，代表生命周期终结)

## 四、 状态流转路径 (State Transition Paths)

### 1. 正常正向流转 (Normal Forward Flow)
这是订单顺利推进的标准路径。

*   **S00 -> S01 (筹备转沟通)**
    *   **触发动作**: 方案/沟通材料就绪，客户可进入意向沟通。
*   **S01 -> S02-01 (意向转提案设计)**
    *   **触发动作**（产品/后台语义）：意向沟通完成、方案确认后进入深化；**当前 C 端已取消「意向报价单」**，意向阶段以方案反馈为主，具体跳转以后台/workflow 为准。
*   **S02-01 -> S02-02 (提案设计转订购报价)**
    *   **触发动作**: 设计师完成深化方案和详细图纸，进入精准报价阶段。
*   **S02-02 -> S03 (订购报价转订购确认)**
    *   **触发动作**: 设计师完成精准报价，在后台点击“提交客户确认”。
*   **S03 -> S06-01 (订购转交付设计)**
    *   **触发动作**: 客户在订单详情页查看“最终报价单”，点击“确认报价单”并完成签名。
*   **S06-01 -> S06-02 (设计转汇报)**
    *   **触发动作**: 设计师完成所有施工图纸，在后台点击“发起方案汇报”。
*   **S06-02 -> S06-03 (汇报转备货)**
    *   **触发动作**: 客户在详情页浏览完所有图纸，点击“完成审阅”并同意方案。
*   **S06-03 -> S06-04 (备货转施工)**
    *   **触发动作**: 施工经理在后台确认材料和人员就位，点击“进场开工”。
*   **S06-04 -> S06-05 (施工转内审)**
    *   **触发动作**: 施工经理完成所有施工节点，在后台点击“发起内部审核”。
*   **S06-05 -> S07 (内审转验收)**
    *   **触发动作**: 内部审核通过，提交竣工验收及最终结算单，等待客户确认。
*   **S07 -> S11 (验收通过交付)**
    *   **触发动作**: 客户在订单详情页查看“订单结算单”，点击“确认结算单”并完成签名。

### 2. 逆向/打回流转 (Reverse / Rejection Flow)
当客户对方案、报价或施工质量不满意时触发的打回路径。

*   **S01 -> S04 (意向转婉拒)**
    *   **触发动作**: 客户对方案或沟通结果不满意，明确表示拒绝。
*   **S03 -> S02-01 / S02-02 (订购打回提案/报价)**
    *   **触发动作**: 客户在“订购确认中”阶段，点击底部的“希望调整方案”并提交反馈。订单退回提案设计或订购报价阶段重新修改。
*   **S06-02 -> S06-01 (方案打回设计)**
    *   **触发动作**: 客户在“方案汇报中”阶段，浏览图纸后提出修改意见，点击“提交反馈”。订单退回交付设计阶段。
*   **S06-05 -> S06-04 (内审不合格打回施工)**
    *   **触发动作**: 内部审核发现问题，打回施工阶段进行整改。
*   **S07 -> S09 (验收不合格整改)**
    *   **触发动作**: 客户在“订单验收中”阶段，对施工质量或结算单有异议，点击“对结算有异议”并提交反馈。
*   **S09 -> S07 (整改完成转验收)**
    *   **触发动作**: 施工团队完成整改，重新提交验收申请和更新后的结算单。

### 3. 异常/超时/终止流转 (Exception / Timeout / Termination Flow)
订单在未正常交付前被提前结束，或因客户未及时操作而进入特殊状态。

*   **S01 -> S05 (意向超时转决策中)**
    *   **触发动作**: S01阶段设定的时间到期，客户迟迟未做反馈，系统自动流转至S05。
*   **任意状态 (S01~S07) -> S08 (订单终止)**
    *   **触发动作**: 客户在订单详情页右上角点击红色的“终止订单”按钮，或后台管理员强制终止。
    *   **说明**: 终止后，订单进入冻结状态，不可再进行常规的正向流转，通常需要走线下清算流程。

### 4. 售后/维保流转 (Post-sales / Maintenance Flow)
订单正常交付后发生的售后服务。

*   **S11 -> S10 (发起售后维保)**
    *   **触发动作**: 客户在“订单已交付”状态下，在详情页右上角点击蓝色的“申请维保”按钮，填写维保需求。
*   **S10 -> S11 (维保完成)**
    *   **触发动作**: 维保团队完成维修服务，客户确认修复完毕，订单状态恢复为“已交付”。

## 五、 财务单据显示逻辑 (Financial Documents Visibility Logic)

为了确保信息流的严谨性和减少用户干扰，系统根据订单所处的阶段和状态，动态控制各类报价单与结算单的可见性。

### 1. 意向阶段与报价单（已调整）
*   **产品规则**：**不再提供「意向报价单」**；意向期通过方案反馈/沟通完成确认，**订购报价单从 V1 起**在订单详情与 `/quotation` 展示（见 `PAGES_GUIDE` §2.15）。
*   **S00/S01/S05**：订单详情「报价单」列表中**不出现**意向类报价条目。

### 2. 订购报价单 (Ordering Quotation)
*   **可见性**: 订单进入 **订购期 (Ordering Phase)** 及后续所有阶段时可见。
*   **逻辑**: 在“意向期”内，不显示任何订购相关的详细报价单。
*   **报价单状态**：客户在S03的时候是还没有签字的，签字之后S03就会跳到S06（交付期），所以在S03阶段，报价单状态应为“已查看已反馈”或“已查看未签字”，进入S06后才变为“已查看已签字”。

### 3. 交付报价单 (Delivery Quotation)
*   **可见性**: 订单进入 **交付期 (Delivery Phase)** 及后续所有阶段时可见。
*   **逻辑**: 在“意向期”和“订购期”内，交付报价单处于隐藏状态。
*   **报价单状态**：交付报价单通常在S06阶段作为施工依据，确认后推进施工。

### 4. 最终结算单 (Settlement Statement)
*   **可见性**: 仅在订单进入 **验收期 (Acceptance Phase)** 及后续阶段（维保、结束）时可见。
*   **逻辑**: 在正式进入验收流程之前，系统不会生成或展示最终结算单。
*   **报价单状态**：客户在S07的时候是还没有签字的，签字之后S07就会跳到S11（已交付），所以在S07阶段，结算单状态应为“未查看未签字”或“已查看未签字”，进入S11后才变为“已结清/已查看已签字”。

## 六、 财务单据状态说明 (Financial Documents Status Specifications)

为了统一用户端显示的文案与视觉反馈，系统定义了以下标准状态提示：





### 1. 报价单与结算单状态生命周期 (Status Lifecycle)

报价单与结算单遵循相同的四种核心状态流转逻辑：

| 状态 | 描述 | 触发动作 (驱动变化) |
| :--- | :--- | :--- |
| **未查看未签字** | 初始状态，单据已生成 | 无 (初始状态) |
| **已查看未签字** | 客户已阅读单据 | 客户打开单据详情页 |
| **已查看已反馈** | 客户已提交修改意见 | 客户点击“希望调整方案”并提交反馈 |
| **已查看已签字** | 客户已确认单据 | 客户点击“确认单据”并完成签名 |

#### 状态流转路径
1.  **未查看未签字** --(客户查看)--> **已查看未签字**
2.  **已查看未签字** --(客户签字)--> **已查看已签字**
3.  **已查看未签字** --(客户反馈)--> **已查看已反馈**
4.  **已查看已反馈** --(设计师/后台更新单据)--> **未查看未签字** (新单据生成)

*注：在“已查看已反馈”状态下，客户无法进行签字操作，必须等待后台根据反馈更新单据后，重新进入新的“未查看未签字”状态。*

### 3. 视觉反馈规范 (Color Coding)

*   **红色 (Danger)**: 用于 `未查看未签字`，提示客户有新单据待处理。
*   **橙色 (Warning)**: 用于 `已查看未签字`，提示客户需尽快完成签字。
*   **绿色 (Success)**: 用于 `已查看已签字`，表示流程已闭环。
*   **蓝色 (Info)**: 用于 `已查看已反馈`，表示单据正在处理反馈意见。
